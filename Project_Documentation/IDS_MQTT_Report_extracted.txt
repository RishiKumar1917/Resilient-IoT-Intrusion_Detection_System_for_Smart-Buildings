Network Intrusion Detection System
for IoT MQTT Traffic
Minor Project Report
Platform: Kali Linux (VM, Bridged Mode)
IDS Tool: Snort 3 (Snort++)
Protocol Focus: MQTT (Port 1883)
Date: May 5, 2026
1. Objective
The objective of this module is to implement a network-based intrusion detection system (IDS) capable of monitoring real-time IoT communication and identifying malicious activities. The system focuses specifically on MQTT-based communication, which is widely used in IoT environments for lightweight data transmission.
2. System Overview
The network monitoring system is designed to observe communication between IoT devices and the MQTT broker. The IDS operates passively, capturing network packets and analyzing them using rule-based detection.
Architecture
IoT Device (ESP32)  →  MQTT Broker (192.168.1.8)  →  Subscriber
                              ↓
                          Snort IDS
                              ↓
                      Alert Generation
MQTT Broker is hosted at 192.168.1.8
IDS (Kali Linux) operates at 192.168.1.22
Both systems connected within the same local network using bridged mode
3. Snort Configuration
Snort 3 (Snort++) was configured in IDS mode to monitor network traffic on the interface eth0.
Execution Command
sudo snort -i eth0 -A alert_fast -c /etc/snort/snort.lua
-i eth0 — specifies the network interface to monitor
-A alert_fast — enables concise, fast alert output format
-c /etc/snort/snort.lua — loads the Snort configuration file
4. Custom Rule Set
A set of custom rules was developed to detect MQTT-related activities and common network attack patterns.
Rule Name
Snort Rule
MQTT Connection Attempt
alert tcp any any -> any 1883 (msg:"[IoT] MQTT Connection Attempt"; sid:2000001; rev:1;)
SYN Packet Detection
alert tcp any any -> any 1883 (flags:S; msg:"[IoT] MQTT SYN Packet (Possible Scan)"; sid:2000002; rev:1;)
High MQTT Traffic
alert tcp any any -> any 1883 (msg:"[IoT] High MQTT Traffic"; detection_filter:track by_src, count 20, seconds 5; sid:2000003; rev:1;)
Ping Detection
alert icmp any any -> any any (msg:"[IoT] Ping Detected"; sid:2000004; rev:1;)
Rule Functionality Details
MQTT Connection Attempt — Detects any TCP connection to port 1883 (MQTT broker)
SYN Packet Detection — Identifies potential port scanning activity via TCP SYN flags
High MQTT Traffic — Detects abnormal traffic volume indicative of flooding or DoS using a detection filter (20 packets in 5 seconds)
Ping Detection — Identifies reconnaissance activity via ICMP echo requests

8. Attack Simulation and Detection
Three types of network attacks were simulated to evaluate the effectiveness of the IDS. Each attack was launched from the Kali Linux machine and Snort alerts were observed in real time.
8.1  Reconnaissance Attack — ICMP Ping Sweep
Command Used:
ping 192.168.1.8
Expected Alert:
[IoT] Ping Detected
This attack simulates an adversary attempting to identify active devices on the network. The IDS successfully detected ICMP echo requests directed at the MQTT broker.
Figure 7: Snort detecting ICMP ping sweep — repeated [IoT] Ping Detected alerts for 192.168.1.22 ↔ 192.168.1.8
Figure 8: Additional ping detection alerts including IPv6 ICMP background traffic
8.2  Port Scanning Attack — TCP SYN Scan
Command Used:
nmap -p 1883 192.168.1.8
Expected Alert:
[IoT] MQTT SYN Packet (Possible Scan)
This attack simulates probing of open ports to identify available services. The IDS detected SYN packets targeting the MQTT port, indicating scanning behavior. The screenshot below shows the transition from ping-based reconnaissance to port scanning.
Figure 9: Snort alerts showing [IoT] MQTT SYN Packet (Possible Scan) and [IoT] MQTT Connection Attempt alongside prior ping detections
8.3  Traffic Flooding Attack — MQTT DoS Simulation
Command Used:
for i in {1..30}; do mosquitto_pub -h 192.168.1.8 -t test -m "attack"; done
Expected Alert:
[IoT] High MQTT Traffic
A burst of MQTT messages was generated to simulate abnormal traffic conditions. The IDS identified this as high-volume traffic using a detection_filter mechanism (threshold: 20 packets in 5 seconds from the same source).
Figure 10: Snort detecting the DoS flood — [IoT] High MQTT Traffic and [IoT] MQTT Connection Attempt alerts in rapid succession
9. Results and Analysis
The system successfully achieved the following outcomes:
Real-time monitoring of MQTT traffic on port 1883
Detection of ICMP-based reconnaissance (ping sweep)
Detection of TCP SYN-based port scanning activity
Detection of high-volume MQTT traffic indicative of a DoS attack
Identification of both normal MQTT communication and anomalous network behaviour
The use of protocol-specific custom rules enabled accurate, real-time detection of IoT-related activities. All three simulated attack types triggered their respective Snort alerts, confirming the effectiveness of the rule set.
10. Limitations
The rules are signature-based and may generate false positives in high-traffic environments
Deep packet inspection for MQTT payload content was not implemented
IPv6 background traffic generated additional ICMP alerts not directly related to simulated attacks
The detection_filter threshold may require tuning for different deployment environments
11. Conclusion
The implemented Snort-based IDS successfully monitored real-time MQTT communication and detected multiple network attack scenarios including reconnaissance, port scanning, and traffic flooding. The system demonstrates the feasibility of integrating network-level security monitoring into IoT environments, providing visibility into both normal and malicious network activities.
The modular rule-based approach allows for easy extension — additional rules can be added to detect MQTT-specific payload anomalies, authentication failures, or other IoT attack vectors as future work.
